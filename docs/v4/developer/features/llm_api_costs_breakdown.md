# Quilltap LLM cost reduction: a prioritized engineering report

**The 31.2M-token session is not a model-choice problem; it is a prompt-architecture problem.** Three structural defects compound into the current spend: (1) summarization re-reads full history from scratch on every turn for every character, (2) the CHAT_MESSAGE prompt is shaped so that every byte downstream of a small dynamic region is uncacheable, and (3) all four providers in the stack — Anthropic, OpenAI, xAI/Grok, and Z.AI/GLM — now support prompt caching at 75–90% read discounts, but Quilltap currently emits zero cache markers and has no provider's automatic prefix machinery firing. Fixing the summarization redundancy alone removes ~95% of token volume; fixing the prompt structure alone removes ~75–85% of dollar cost on the expensive CHAT_MESSAGE calls. **Done together, on a session of this shape, the realistic projected reduction is ~85–92% of total spend** with no quality regression and the per-character perspective semantics fully preserved. The remainder of this report explains why, then lists the fixes ordered by ROI.

## How current spend decomposes and why

Two unrelated waste mechanisms account for the vast majority of cost. **SUMMARIZATION (252 calls, 26.2M tokens, 86% of token volume)** wastes tokens because each of three per-turn calls dumps the full conversation history to gpt-5-nano even though only one new turn was added since the last call. The 0.146 turn-to-turn Jaccard similarity in outputs is the smoking gun: the LLM is reconstructing the same world state from scratch every turn rather than extending a prior summary. At 84 turns, ~97% of every from-scratch summarization call is recomputing already-summarized content. This is the exact failure mode Factory.ai documents as "redundant re-summarization" and the qvink/SillyTavern-MessageSummarize maintainer warns is "prone to inaccuracies and missing details … one bad generation can completely ruin it."

**CHAT_MESSAGE (~81 calls, ~2.9M tokens) is where dollars concentrate** because the targets are Sonnet, GPT-5.3-chat-latest, GLM-5.1, and Gemini 2.5 Flash. Three independent failure modes stack to produce 0% cache hit rate: a stable suffix of 3.7K–3.9K characters appears *after* the dynamic region (the prefix hash never reaches it); character d9d0d998 has a 1-character stable prefix indicating dynamic content prepended to its system block (timestamp, request ID, randomized macro, or non-deterministic plugin iteration); and history append-only-ness drops to 5–78% on three of four characters, indicating mid-history rewrites that invalidate every cache entry downstream of the rewrite. **Memory whispers re-rendered as user-role messages with rank-driven ordering are the most likely root cause of the mid-history reshuffling** — every semantic re-rank shifts whisper indices, and every shift breaks the cache from that index forward.

The CONTEXT_COMPRESSION step firing 1× in 84 turns is the third symptom: it is gated as a hard ceiling rather than a working threshold, so eager per-turn re-summarization is being done in the SUMMARIZATION pipeline instead of being properly compaction-driven.

## Provider caching mechanics, verified to April 2026

All four providers now support prompt caching, with the same fundamental contract — static-prefix-first, dynamic-suffix-last, byte-identical — but with materially different mechanics. The single architectural decision is to design the prompt for Anthropic's stricter explicit-breakpoint API, which automatically wins on all four.

**Anthropic** offers two modes as of 2025–2026. Top-level `cache_control: { type: "ephemeral" }` enables **automatic caching** that walks a breakpoint forward as the conversation grows; up to **4 explicit breakpoints** can be placed on individual content blocks within `tools`, `system`, and `messages.content`. The cache hierarchy is **tools → system → messages**, cumulative; any change at a level invalidates that level and everything after. Critically, the system performs a **20-block lookback** from each breakpoint, so a fast-growing conversation that adds many tool_use/tool_result blocks per turn must place a second explicit breakpoint earlier or it will silently miss cache. Two TTLs: **5-minute (1.25× input write cost) refreshed for free on every hit, and 1-hour (2× write cost)**; **cache reads are 0.10× base input** — a 90% discount. Minimum cacheable prefix is **2,048 tokens for Sonnet 4.6, 1,024 tokens for Sonnet 4.5/4**. As of Feb 5, 2026, caches are **workspace-isolated** on the Claude API and Azure (still org-isolated on Bedrock/Vertex).

**OpenAI** caching is automatic and prefix-based with a **1,024-token minimum** and 128-token hit increments. Two new 2025–2026 parameters matter: `prompt_cache_key` (combines with the prefix hash for sticky routing; keep each key under ~15 RPM) and `prompt_cache_retention: "in_memory" | "24h"` (extended retention is supported on gpt-5.5/5.5-pro/5.2/5.1 family but **gpt-5.3-chat-latest is not in the extended-retention list — assume in-memory only**). Cache reads are **typically 0.10× input (90% off) on the GPT-5.x line**, but **GPT-5.5 specifically uses 0.25× (75% off)** per the live pricing page; verify per model. For gpt-5.3-chat-latest at $1.75 input / $0.175 cached, a 30K-input call drops from $0.060 to $0.015 on a hit (~74% reduction). Tools and structured-output schemas count toward the cacheable prefix and the 1,024-token minimum.

**xAI/Grok supports prompt caching** as of 2026 — automatic prefix caching across all `grok` language models, with **the critical caveat that cache entries are per-server**. Without a stable conversation key, requests round-robin and miss. On Chat Completions, set the **`x-grok-conv-id` header** to a stable per-session UUID; on the Responses API endpoint (`https://api.x.ai/v1/responses`), use the **`prompt_cache_key` body field** (passed via `extra_body` in the OpenAI SDK). Discounts are 75–90% by model (grok-4.20-reasoning: $2.00 → $0.20 input, 90% off; grok-4.1-fast: $0.20 → $0.05, 75% off). No published minimum prefix size; TTL is best-effort with no SLA. Grok's own migration guide endorses the OpenAI SDK Responses API path the user is already planning.

**Z.AI/Zhipu/GLM supports "Context Caching"** automatically across GLM-4.5+ and the GLM-5/5.1 line — no `cache_control`, no breakpoints, just byte-identical prefix matching. **GLM-5.1 input drops from $1.40 to $0.26 per million on cache hits** (~81% off); GLM-4.7 from $0.60 to $0.11 (~82% off). Response shape is OpenAI-compatible (`usage.prompt_tokens_details.cached_tokens`). Minimum prefix size and TTL are not published. The official docs page gives a "50% off" illustrative example that materially understates the actual per-model discount; trust the live pricing page (~75–82% off) instead.

**The implication is that all four providers reward the same prompt shape**: a long, stable, sorted, canonicalized prefix of `tools → system_global → character_persona → memory_pool → frozen_history`, with all dynamic content (timestamps, identity reinforcement, scene-state deltas, memory rank instructions, the new user message) concatenated into a single tail user message. Anthropic gets explicit breakpoints; OpenAI/Grok/Z.AI inherit the structural win automatically.

### Per-character cache identifier (2026-05)

The provider cache identifier (`prompt_cache_key` for OpenAI/Grok, `user_id` for DeepSeek V4+, `user` field for OpenAI-compatible / Z.AI / OpenRouter) is keyed on `characterId`, not `chatId`. The persona block (manifesto / identity / description / personality) sits high in the prompt, so each speaker rotation in a group chat already busts the cacheable prefix — keying by chat buys nothing. Keying by character means the persona block becomes the actual cache prefix and the same character shares a warm cache across chats.

**Multi-character group chats keep N parallel caches by design.** Each character in a group chat gets its own cache namespace. The cache-hit discount (~10× on Anthropic, ~120× on DeepSeek V4-Pro, 75–90% on OpenAI/Grok) trivially justifies running parallel caches per character — this is precisely the workload the feature is designed for.

The single Quilltap-side cache identifier (`LLMParams.cacheKey`) is built in [`lib/llm/cache-key.ts`](../../../lib/llm/cache-key.ts) by `buildCharacterCacheKey(characterId)` → `quilltap:char:<characterId>:v<n>`. The version constant `PROMPT_CACHE_STRUCTURE_VERSION` (currently 2) is bumped whenever the cacheable prompt structure changes intentionally (tool-schema shape, system-prompt builder layout, persona-block format, memory-pool format). Bumping forces a clean cold-cache rollover at every provider on first use after deploy.

Plugins apply the identifier differently per their provider's mechanics — OpenAI/Grok use it as a sticky routing hint, DeepSeek as a KV-cache isolation namespace, OpenAI-compatible plugins forward it as `user`, Anthropic ignores it (content-hashed `cache_control` is orthogonal). See [docs/developer/features/complete/per-character-prompt-caching.md](complete/per-character-prompt-caching.md) for the full design.

## Priority 1: redesign summarization (the 86% of token volume)

The current "three from-scratch summaries per turn" pattern must be replaced with a **two-stage shared-base + per-character-filter** architecture. One incremental base summary is generated per compaction event by feeding the prior base summary plus only the new turns since last update. Each character's summary then becomes a **cheap perspective filter** over that base summary: a deterministic visibility filter using explicit witness sets, followed by a short LLM rewrite for voice. This preserves the load-bearing per-character knowledge differentiation — character-joined-at-turn-30 will literally never see facts tagged with witness sets that exclude them — while collapsing summarization input from ~104K tokens/call × 252 calls/session to ~4K tokens × ~10 base updates plus ~1.5K tokens × ~3 per-turn filter calls.

Witness tracking must be **explicit and deterministic**, not LLM-inferred. Every event gets `witnessedBy: Set<CharId>` set at write time from Quilltap's existing scene-membership state. The base summary preserves witness tags inline (e.g., `[w:alice,bob] Alice and Bob entered the tavern. [w:alice] Alice noticed a hidden compartment.`) so the filter step can drop forbidden lines deterministically before the LLM ever sees them. The Theory-of-Mind multi-agent literature (arXiv 2310.10701) shows explicit belief states beat asking the LLM to re-infer who knew what, and PerSphere (arXiv 2412.12588) validates that a shared retrieval/extraction stage feeding per-perspective summarizers outperforms monolithic generation.

**Triggering must shift from per-turn to budget-driven.** The current CONTEXT_COMPRESSION-as-hard-ceiling pattern fires once in 84 turns precisely because eager re-summarization elsewhere prevents it from ever reaching the ceiling. Adopt a triple-gate matching Anthropic's `compaction_control` and Gemini CLI's `COMPRESSION_TOKEN_THRESHOLD = 0.7`: a soft refresh trigger at 8K rolling-buffer tokens or 8 turns since last refresh; a layer-promotion trigger when the working summary layer hits 20 snippets; and a hard full-rebuild trigger every 50 turns or on drift detection, to reset accumulated recursive error (MemGPT's recursive-baseline degraded to 35.3% accuracy after many compactions, motivating periodic rebuilds). Most turns should `return "skip"` and reuse cached per-character summaries unchanged — expected hit rate of incremental work is ~1 in 8 turns rather than 3-of-3 every turn.

**Token math for an 84-turn session.** Current pattern: 252 calls × ~104K tokens average = 26.2M input tokens. Two-stage incremental: ~10 base updates × 4K + ~30 per-character filter calls × 1.5K + skipped turns reusing cached summaries ≈ **85K tokens — a 99.7% reduction in summarization input volume**. Even with a 10× safety factor on call counts and sizes, the reduction floor is 95%. At gpt-5-nano pricing, summarization cost drops from ~$1.31/session to under $0.01/session. The bigger downstream win, however, is that the per-character summaries become small, stable artifacts that the CHAT_MESSAGE prompt can cite directly rather than reconstructing context inline.

Implementation should follow the qvink/SillyTavern-MessageSummarize anchoring pattern: every summary is attached to its source-message anchor IDs, so editing message *m* invalidates only summaries that included it. This makes summary maintenance edit-aware and allows partial re-runs rather than full rebuilds on user corrections. The Summaryception layered-recursion design (5 layers, doubling coverage per layer, seed-promotion without LLM call on first promotion to a new layer) keeps per-turn summarization cost **O(1) regardless of conversation length**, with published numbers showing 9,307 turns compressed into 16K tokens at cranked settings.

## Priority 2: restructure the CHAT_MESSAGE prompt for cacheability

The recommended canonical structure is a **four-tier hierarchy with explicit Anthropic breakpoints, where each tier matches a distinct change frequency**: tools (1h TTL, sorted alphabetically by name with canonicalized JSON Schema, breakpoint on the last tool entry); a global system block of roleplay framework and format rules (1h TTL); a per-character persona block (5m TTL); a hash-stable memory pool (5m TTL). Conversation history follows as a pure projection of an append-only event log, and **all dynamic content is concatenated into a single tail user message** containing — in this order — identity reinforcement wrapped in `<scene-state>` tags, scene-state delta, memory rank instruction, then the new user message. The top-level `cache_control: { type: "ephemeral" }` provides a fifth de-facto auto-managed breakpoint at the last messages block, walking forward as history grows.

This structure addresses every observed defect simultaneously. The 3.7–3.9K-character stable suffix on characters 3b476cd1 and e06c5f31 disappears because nothing exists after the user message. The d9d0d998 1-char prefix bug is fixed by introducing a deterministic system-block builder with a CI test asserting `expect(builder.build(input)).toEqual(builder.build(input))` and a stability snapshot — the most likely culprits are a timestamp ("Current time: 2026-04-29T…"), a per-session UUID prepended at top, a `{{random}}` macro, or non-deterministic plugin-iteration order from a Map/Set without explicit `.sort(orderKey)`. Mid-history reshuffling on bbf1715a (50–78% append-only) and d9d0d998 (5–66%) is fixed by treating the chat log as an **append-only event log** with a pure materialized-view function `f(events, asOfTs) → messages[]` that is **temporally monotonic** — `f(events, t1) ⊆ f(events, t2)` at every index that existed at t1 — and forbids mid-history mutation outside the rolling tail window of the last K=20 turns. Retroactive scene-state corrections become new events appended at the tail ("[Retroactive note: at turn 12, Alice had already joined.]"), never inserted at index 12.

**The memory-whisper re-ranking problem requires a two-tier retrieval restructuring.** Replace the "3 whispers × 11–14K chars persisted as monotonically growing user-role messages" pattern with: (1) a single **stable memory pool** in the system tier, sorted by `memory_id` (never by rank), capped at ~30 memories, frozen at session start and refreshed only at compaction boundaries; (2) a tiny **rank instruction in the tail** (~50 tokens: "Most relevant memories for this turn: m_3a4f, m_91be") that biases attention without disturbing the cached pool. The pool becomes byte-identical across turns regardless of how the rank changes; only the 50-token tail varies. This is the pattern used by Cursor, RAGCache (ACM TOCS 2025), and described in the "Beyond Prompt Caching" pipeline literature. Optionally, repeat the top-1 memory's text inline in the rank instruction (uncached but tiny) so the model doesn't ignore the highest-priority memory.

Tool definitions, the most stable thing in the prompt, must lead with deterministic JSON serialization. **Sort by `name`, canonicalize JSON Schema with sorted keys, never insert/remove tools mid-session.** Anthropic's last tool entry gets `cache_control: { type: "ephemeral", ttl: "1h" }`. With 34 tools at ~3.5K tokens redundantly retransmitted across 970 calls, the uncached cost is ~$10/session on Sonnet alone; with caching, it collapses to $1.

**Quantitative ceiling for Quilltap CHAT_MESSAGE.** Of a 30K-token mid-conversation request, ~28K tokens become cacheable (tools 4K + system 4.5K + memory pool 5K + frozen history 14.5K) and ~2K tokens remain dynamic (tail user message). On Sonnet 4.5 a cache hit costs $0.0144 vs $0.0900 uncached — **84% reduction per call**. Across 81 CHAT_MESSAGE calls in a typical session, with one cache write per ~10-minute idle window plus continuous refreshes on hits, projected input cost drops from ~$7.29 to ~$1.94 (73% reduction conservative) or ~$1.26 (83% reduction if calls are tightly cadenced). Anthropic publishes up to 85% TTFT reduction on long cached prompts, materially benefiting the chat UX as well.

## Priority 3: per-provider implementation specifics

The plugin layer should expose a single canonical `buildRequest(ctx, plugins)` that emits the four-tier structure, then per-provider adapters apply provider-specific caching primitives. **For Anthropic, place explicit `cache_control` on the last tool, on each system block (tier-appropriate TTL), and add top-level `cache_control: { type: "ephemeral" }` for auto-managed messages.** Use the new `extended-cache-ttl-2025-04-11` beta-equivalent (now GA, no header needed) for 1-hour TTL on tools and global system. Plan for the 20-block lookback by ensuring the active breakpoint moves forward at most every ~15 turns; if Quilltap fires many tool_use/tool_result blocks per turn, add a second mid-history explicit breakpoint to stay inside the lookback window.

**For OpenAI, the prompt structure is sufficient on its own — caching is automatic — but two parameters materially affect hit rate.** Set `prompt_cache_key: \`quilltap:character:${characterId}:v${projectVersion}\`` to pin sticky routing; without it, requests above ~15 RPM/key shard across machines and degrade hit rate. Versioning the key by `projectVersion` ensures clean cutover when tool schemas or system prompts change. For gpt-5.3-chat-latest, omit `prompt_cache_retention` (extended retention not in the supported model list per current docs); for gpt-5.1/5.2/5.5 family, set `"24h"` for sessions that span coffee breaks. **For ZDR users, force `"in_memory"` retention** — extended caching persists KV tensors to GPU-local storage and is not ZDR-eligible — and avoid gpt-5.5 (no in-memory option) for those users.

**For xAI/Grok, the migration to OpenAI SDK Responses API is the right move and unlocks caching simultaneously**, provided `prompt_cache_key` is passed via `extra_body` in every call and set to a stable per-session UUID. Without it, Grok's per-server cache architecture round-robins requests and every turn misses. The cached-token field renames from `usage.prompt_tokens_details.cached_tokens` (Chat Completions) to `usage.input_tokens_details.cached_tokens` (Responses API); update logging accordingly. For non-real-time work (offline summarization, embedding prep), Grok's Batch API stacks 50% off all token types on top of caching.

**For Z.AI/GLM, no caching code is needed** — Context Caching is fully automatic — but the prompt structure must be exact-prefix-stable since the docs' "highly similar" language is best treated as byte-exact. Verify cache hits by inspecting `usage.prompt_tokens_details.cached_tokens` on the OpenAI-compat endpoint. **GLM-4.7 at $0.60 input / $0.11 cached deserves consideration as a primary or fallback model**: it was Z.ai's flagship until weeks before GLM-5.1 launched, retains very close chat-quality, and costs less than half. GLM-4.5-Flash is free across all token types and is suitable for auxiliary tasks (intent detection, classification, the per-character filter step in the new summarization pipeline).

## Risks the rollout must manage

The largest behavioral risk is **moving identity reinforcement out of the system block**. Anthropic-trained models give system-role text more authority than user-role text, and a tail-side "you are now Alice" may be obeyed less faithfully than a system-block injection. Fix #14 (hybrid identity reinforcement: minimal generic system claim plus dynamic tail binding) materially reduces this risk; the eval harness must measure character-voice consistency and tool-call correctness across a stable benchmark of 50–100 representative turns with multiple swipes.

The second risk is **cache invalidation cascades** at deploy time. Tool-schema changes, persona edits, and `projectVersion` bumps blow away every cache below them and produce a thundering herd of cache writes. Mitigations: roll out tool-schema changes at off-peak; pin `prompt_cache_key` and Anthropic workspace by `projectVersion` so versions never share caches; never hot-reload tools mid-session.

The third risk is **summarization drift over many compactions**. MemGPT's pure recursive-summary baseline degraded to 35.3% accuracy. The architecture mitigates this via witness-tagged fact pinning (tagged lines survive re-summarization verbatim), capped recursion depth (≤5 layers), and full-rebuild every 50 turns. Edit-aware invalidation via summary-anchor IDs ensures user edits don't cascade through the entire summary chain.

A final operational risk is **per-character cache lineage in group chats**. With multiple characters sharing a scene, two patterns are viable: separate `prompt_cache_key` per character (cache resets on every speaker change but persona blocks are clean), or all personas in one system block with the active speaker indicated only in the tail (full system stays cached, only tail varies on speaker switch). Option two is preferred when characters share a scene continuously; option one when isolation matters more than cache reuse.

## What this unlocks

Combining these fixes converts a session from a $5–10 burn into a $0.50–1.50 burn at the same conversational quality and the same per-character knowledge guarantees. **Summarization volume drops 95–99%**; **CHAT_MESSAGE input cost drops 73–84%** once the cache is warm; **tool definitions go from $10 of redundancy to $1** on Sonnet; and **TTFT improves by 50–67% on long cached prompts**, which materially improves the desktop chat UX. None of this requires giving up the perspective-shaped per-character summaries, switching off expensive models, or reducing context. The shape of the fix is structural, not a quality trade.

The biggest non-cost upside is **the architecture becomes debuggable**. With per-tier prefix hashes logged on every call and a `/cache-diag` endpoint that re-runs the prompt builder against the last 10 turns of any session, the d9d0d998-style bug becomes a 30-second triage rather than the multi-week diagnostic that produced this report's input data. The append-only event log makes user edits, retroactive scene corrections, and character joins/leaves first-class events that materialize identically every turn — eliminating an entire class of latent state-mutation bugs that currently manifest as cache misses.

## Conclusion

Quilltap's spend is dominated by two fixable architectural defects: a per-turn, per-character full-history summarization loop that wastes ~95% of its tokens on rework, and a CHAT_MESSAGE prompt whose stable content is sandwiched between dynamic regions in a way that defeats every provider's cache. Both defects are fixable at the plugin layer with no model swaps, no quality compromise, and no loss of per-character perspective semantics. The per-character knowledge model — the load-bearing feature — actually becomes *stronger* under the recommended architecture because witness sets become explicit data rather than implicit LLM inference.

The sharpest single insight is that **all four providers in Quilltap's stack now reward exactly the same prompt shape**: long stable prefix, single dynamic tail. Designing once for Anthropic's stricter explicit-breakpoint API yields automatic wins on OpenAI's prefix matcher, Grok's per-server cache (with `prompt_cache_key`), and Z.AI's Context Caching simultaneously. The plugin architecture is precisely the right place to enforce this contract, with provider adapters applying provider-specific primitives (`cache_control` markers for Anthropic, `prompt_cache_key` for OpenAI/Grok, nothing for Z.AI) over a single canonical request shape.
